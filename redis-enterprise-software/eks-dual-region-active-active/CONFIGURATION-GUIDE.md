# Configuration Guide

**Understanding the Two-Layer Architecture**

---

## 🏗️ Architecture Overview

This project uses a **two-layer architecture** to separate infrastructure from application:

```
┌─────────────────────────────────────────────────────────────┐
│                    LAYER 1: INFRASTRUCTURE                  │
│                         (Terraform)                         │
├─────────────────────────────────────────────────────────────┤
│  • VPCs, Subnets, Security Groups                          │
│  • EKS Clusters, Node Groups                               │
│  • VPC Peering, Route53                                    │
│  • IAM Roles, Policies                                     │
│                                                             │
│  Configuration: terraform.tfvars                           │
│  Tool: terraform                                           │
└─────────────────────────────────────────────────────────────┘
                            ↓
              (terraform apply generates)
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                   LAYER 2: APPLICATION                      │
│                    (Shell + Kubernetes)                     │
├─────────────────────────────────────────────────────────────┤
│  • Redis Enterprise Operator                               │
│  • Redis Enterprise Clusters (REC)                         │
│  • Active-Active Database (CRDB)                           │
│  • Monitoring, Backups                                     │
│                                                             │
│  Configuration: post-deployment/config.env (auto-generated)│
│  Tool: kubectl + shell scripts                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 📁 Configuration Files

### **1. terraform.tfvars** (You Edit This)

**Purpose:** Define infrastructure parameters  
**Format:** HCL (HashiCorp Configuration Language)  
**Tool:** Terraform

**Key Variables:**
```hcl
project_prefix = "acme"           # Naming prefix for all resources
region1        = "us-east-1"      # Primary AWS region
region2        = "us-east-2"      # Secondary AWS region
aws_profile    = "default"        # AWS CLI profile
owner          = "your_name"      # Resource owner tag
```

**What it controls:**
- AWS resource names (VPCs, EKS clusters, etc.)
- Network configuration (CIDR blocks, subnets)
- Instance types and sizes
- Tags and metadata

---

### **2. post-deployment/config.env** (Auto-Generated)

**Purpose:** Application deployment configuration  
**Format:** Bash environment variables  
**Tool:** Shell scripts + kubectl

**⚠️ DO NOT EDIT MANUALLY** - This file is generated by Terraform

**Key Variables:**
```bash
export PROJECT_PREFIX="acme"                    # From terraform.tfvars
export REGION1_CLUSTER_NAME="acme-r1-rec-..."  # From Terraform output
export REGION2_CLUSTER_NAME="acme-r2-rec-..."  # From Terraform output
export CRDB_NAME="acme-crdb-production"        # Database name
export CRDB_MEMORY="150GB"                     # Database memory
```

**What it controls:**
- Kubernetes deployment scripts
- Redis database configuration
- Monitoring and backup settings
- DNS and networking for Redis

## 🔄 The Workflow

### **Step 1: Edit terraform.tfvars**
```bash
vim terraform.tfvars
```
Set your `project_prefix`, regions, and AWS settings.

### **Step 2: Run terraform apply**
```bash
terraform apply
```
This:
1. Creates AWS infrastructure
2. **Auto-generates** `post-deployment/config.env`
3. Outputs next steps

### **Step 3: Validate Configuration**
```bash
./validate-config.sh
```
This checks that both files are in sync.

### **Step 4: Deploy Application**
```bash
cd post-deployment
source config.env
./deploy-all.sh
```
This deploys Redis using the auto-generated configuration.

## 🎯 Why Two Files?

### **Separation of Concerns**

| Aspect | terraform.tfvars | config.env |
|--------|------------------|------------|
| **Layer** | Infrastructure | Application |
| **Language** | HCL | Bash |
| **Tool** | Terraform | kubectl + scripts |
| **Lifecycle** | Long-lived | Regenerated on each apply |
| **Edit?** | ✅ Yes | ❌ No (auto-generated) |

### **Benefits**

1. **Single Source of Truth:** `terraform.tfvars` drives everything
2. **No Manual Sync:** Terraform generates `config.env` automatically
3. **Type Safety:** Terraform validates infrastructure config
4. **Flexibility:** Can customize application layer independently
5. **Reusability:** Same pattern works for multiple environments

## 🔧 Customization

### **Infrastructure Changes**
Edit `terraform.tfvars` and run `terraform apply`:
```hcl
# Example: Change instance type
node_instance_type = "m5.2xlarge"  # Instead of m5.xlarge
```

### **Application Changes**
Edit the Terraform `local_file` resource in `main.tf`:
```hcl
resource "local_file" "config_env" {
  content = <<-EOT
    export CRDB_MEMORY="200GB"  # Increase database memory
  EOT
}
```
Then run `terraform apply` to regenerate `config.env`.

## ✅ Validation

The `validate-config.sh` script ensures:

1. ✅ Both files exist
2. ✅ `project_prefix` matches in both files
3. ✅ All required variables are set
4. ✅ `config.env` is fresh (recently generated)

---

## 💡 Best Practices

1. **Always run `terraform apply` first** - It generates the config
2. **Never edit `config.env` manually** - Changes will be overwritten
3. **Use `validate-config.sh`** - Before running deployment scripts
4. **Keep `terraform.tfvars` in sync** - Across team members
5. **Version control** - Commit `terraform.tfvars.example`, not `terraform.tfvars`

---

## 🆘 Troubleshooting

**Q: config.env has wrong values**  
**A:** Run `terraform apply` to regenerate it

**Q: Validation fails**  
**A:** Check that `project_prefix` in `terraform.tfvars` is correct

**Q: Can I customize database settings?**  
**A:** Yes, edit the `local_file` resource in `main.tf` and run `terraform apply`
