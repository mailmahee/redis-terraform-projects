# Automated Workflow Guide

**Seamless Infrastructure-to-Application Deployment**

---

## 🎯 Overview

This project now features a **fully automated workflow** where Terraform generates all necessary configuration for the application layer. You only need to edit **one file** (`terraform.tfvars`), and everything else is automated!

---

## 🚀 The Complete Workflow

### **Visual Flow**

```
┌──────────────────────┐
│  1. Edit             │
│  terraform.tfvars    │  ← YOU EDIT THIS (only file you touch!)
└──────────┬───────────┘
           │
           ↓
┌──────────────────────┐
│  2. terraform apply  │  ← Creates infrastructure + generates config.env
└──────────┬───────────┘
           │
           ├─→ Creates AWS Resources (VPCs, EKS, etc.)
           │
           └─→ Generates post-deployment/config.env (automatic!)
           │
           ↓
┌──────────────────────┐
│  3. validate-config  │  ← Validates everything is in sync
└──────────┬───────────┘
           │
           ↓
┌──────────────────────┐
│  4. deploy-all.sh    │  ← Deploys Redis using auto-generated config
└──────────────────────┘
```

## 📝 Step-by-Step Commands

### **Step 1: Configure (One-Time Setup)**

```bash
cd redis-enterprise-software/eks-dual-region-active-active

# Copy example configuration
cp terraform.tfvars.example terraform.tfvars

# Edit with your values
vim terraform.tfvars
```

**Minimum required changes:**
```hcl
project_prefix = "acme"        # Your company/project name
owner          = "your_name"   # Your name
aws_profile    = "default"     # Your AWS CLI profile
```

---

### **Step 2: Deploy Infrastructure**

```bash
# Initialize Terraform (first time only)
terraform init

# Review what will be created
terraform plan

# Deploy infrastructure
terraform apply
```

**What happens:**
1. ✅ Creates VPCs in both regions
2. ✅ Creates EKS clusters
3. ✅ Sets up VPC peering
4. ✅ Configures Route53
5. ✅ **Auto-generates `post-deployment/config.env`** ← Magic happens here!

**Output:**
```
Apply complete! Resources: 101 added, 0 changed, 0 destroyed.

Outputs:

next_steps = <<EOT
✅ Deployment Complete!

✅ Configuration file generated: post-deployment/config.env

Next steps:
1. Validate configuration: ./validate-config.sh
2. Deploy Redis Active-Active: cd post-deployment && source config.env && ./deploy-all.sh
EOT
```

---

### **Step 3: Validate Configuration**

```bash
# Run validation script
./validate-config.sh
```

**What it checks:**
- ✅ `terraform.tfvars` exists
- ✅ `config.env` exists and is fresh
- ✅ `project_prefix` matches in both files
- ✅ All required variables are populated
- ✅ Cluster names are correct

**Expected output:**
```
==========================================================================
  Configuration Validation
==========================================================================

📋 Checking configuration files...

1. Validating PROJECT_PREFIX...
   ✅ MATCH: terraform.tfvars (acme) = config.env (acme)

2. Validating required variables...
   ✅ PROJECT_PREFIX = acme
   ✅ AWS_REGION1 = us-east-1
   ✅ AWS_REGION2 = us-east-2
   ✅ REGION1_CLUSTER_NAME = acme-r1-rec-us-east-1
   ✅ REGION2_CLUSTER_NAME = acme-r2-rec-us-east-2
   ...

3. Checking config.env freshness...
   ✅ config.env is fresh (generated recently)

==========================================================================
✅ VALIDATION PASSED!
==========================================================================
```

---

### **Step 4: Deploy Application**

```bash
# Navigate to post-deployment directory
cd post-deployment

# Load the auto-generated configuration
source config.env

# Deploy everything
./deploy-all.sh
```

**What gets deployed:**
1. ✅ Redis Enterprise Operator
2. ✅ Redis Enterprise Clusters (both regions)
3. ✅ Active-Active Database (CRDB)
4. ✅ Monitoring (Prometheus)
5. ✅ Automated backups
6. ✅ Redis monitoring UI

## 🎨 Key Features

### **1. Single Source of Truth**
- Edit only `terraform.tfvars`
- Everything else is derived automatically

### **2. No Manual Sync**
- `config.env` is generated by Terraform
- Always in sync with infrastructure

### **3. Validation Built-In**
- `validate-config.sh` ensures consistency
- Catches errors before deployment

### **4. Idempotent**
- Run `terraform apply` multiple times safely
- `config.env` is regenerated each time

## 🔄 Making Changes

### **Scenario 1: Change Project Name**

```bash
# 1. Edit terraform.tfvars
vim terraform.tfvars
# Change: project_prefix = "newname"

# 2. Apply changes
terraform apply

# 3. Validate
./validate-config.sh

# 4. Re-deploy if needed
cd post-deployment && source config.env && ./deploy-all.sh
```

### **Scenario 2: Change Database Settings**

```bash
# 1. Edit main.tf (local_file resource)
vim main.tf
# Change: export CRDB_MEMORY="200GB"

# 2. Regenerate config.env
terraform apply

# 3. Validate
./validate-config.sh

# 4. Re-deploy database
cd post-deployment/01-active-active-crdb && ./deploy-crdb.sh
```
